## 💬 面试常见问题 & 标准答案

### Q1: 为什么用规则引擎而不是纯 LLM？

**A**: "医疗场景对安全性要求极高。规则引擎的优势是：
1. **可解释性强**：每条规则都有明确的触发条件和输出
2. **稳定可控**：不会因为 LLM 的随机性产生危险建议
3. **响应速度快**：规则匹配是毫秒级，LLM 调用是秒级
4. **监管友好**：规则可以由医学专家审核，符合医疗器械监管要求

LLM 的作用是增强用户体验：自然语言理解、生成通俗易懂的解释、智能追问。两者结合，既安全又好用。"

---

### Q2: 如果 LLM 返回错误建议怎么办？

**A**: "我们采用了规则优先策略：
1. **风险等级由规则引擎决定**，LLM 无权修改
2. **LLM 只负责解释和建议生成**，且会在 Prompt 中明确约束：
   - HIGH 必须包含'立即就医'
   - MEDIUM 必须建议'24-48小时内就医'
   - LOW 可建议观察但需告知何时应就医
3. **后端会验证 LLM 输出**，如果不符合规则约束，会触发降级机制
4. **全局异常处理**确保无论 LLM 返回什么，系统都能给出安全的兜底建议"

---

### Q3: 追问系统是如何设计的？

**A**: "追问系统有三个核心设计：

**1. 触发条件**：
- 风险等级为 MEDIUM 或 LOW（HIGH 不追问，直接建议就医）
- 症状信息不完整（如缺少持续时间、严重程度）
- 未达到最大追问轮数（3 轮）

**2. 问题生成**：
- 由 Claude 基于当前症状生成 1-2 个关键问题
- 分为 4 类：time（时间）、severity（严重程度）、location（部位）、accompanying（伴随症状）
- 支持选项式（如'前额/两侧/整个头部'）和文本式（开放问答）

**3. 重新评估**：
- 合并原始输入和追问答案，作为新的输入
- 重新执行完整分析流程
- 风险等级可能升级或降级，取决于新信息

这模仿了真实医生的诊断流程：先问主诉，再追问细节，最后综合判断。"

---

### Q4: 如何保证系统的可扩展性？

**A**: "项目采用了清晰的模块化架构：

**后端**：
- `core/models.py`：Pydantic 数据模型，易于扩展字段
- `core/risk_engine.py`：规则引擎独立模块，添加规则只需修改这一个文件
- `core/llm_service.py`：LLM 调用封装，支持切换不同模型
- `core/session_manager.py`：会话管理，可轻松切换 Redis/数据库

**前端**：
- 组件化设计，每个 UI 块都是独立组件
- `api.js` 统一封装 API 调用
- Tailwind CSS 配置化，改色系只需修改 `tailwind.config.js`

**未来可扩展方向**：
1. 接入医学知识图谱，提升规则覆盖度
2. 集成语音输入（Whisper API）
3. 多语言支持（i18n）
4. 用户登录 + 病历管理
5. 对接医院预约系统"

---

### Q5: 项目开发过程中遇到的最大挑战？

**A**: "最大挑战是平衡 AI 的灵活性和医疗的安全性。

**问题**：纯 LLM 方案在测试中出现过几次'LOW 风险'的误判，比如用户描述'胸痛+呼吸困难'，LLM 因为语气平淡就判定为 LOW。

**解决方案**：
1. 设计规则优先架构：风险等级由规则引擎决定，LLM 无权修改
2. 在 Prompt 中加入强约束：明确 HIGH 的判定标准和建议话术
3. 实现三层降级：即使 LLM 完全失败，系统也能返回保守安全的建议
4. 测试 15 个 demo 案例，确保每个都符合医学常识

最终实现了'既智能又安全'的平衡：LLM 提供流畅的用户体验，规则引擎保证医疗安全。"
